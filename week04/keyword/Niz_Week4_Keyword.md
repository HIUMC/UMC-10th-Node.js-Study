# Niz_Week4_Keyword

**ES (ECMA SCript)**

= JavaScript의 문법과 동작을 정의하는 공식 표준 사양(ECMA-262)

- 문제점: 과거 브라우저마다 자바스크립트를 해석하는 방식이 달라 개발자들이 고통받음
- 해결점: 현재 이를 해결하기 위해 만들어진 '자바스크립트 규칙서’임

**ES6**

- ES6의 주요 변화 및 특징
  : 2015년에 발표된 6번째 버전으로, 이전 버전(ES5 이하)에서 문제가 되었던 부분들을 대거 해결함<br>

→ **변수 선언 방식의 개선 (`let`, `const`):** <br>
(과거) 재선언과 재할당이 자유로워 예기치 않은 버그를 만들던 `var` 대신,<br>
(해결) 가변 변수인 `let`과 불변 변수인 `const`가 도입됨.

→ **화살표 함수 (Arrow Function):** <br>
`() => {}` 형태로 코드를 간결하게 작성할 수 있게 됨. 익명 함수로만 만들 수 있으며,<br>
자신만의 `this`나 `arguments`를 가지지 않아 생성자 함수(`new` 연산자)로 사용할 수 없음.

- ES6를 중요시 하는 이유
  : 가독성과 유지 보수 측면에서 혁신적인 향상을 가져온 업데이트이기 때문

**ES Module (ESM)**

= ES6와 함께 도입된 자바스크립트의 공식 모듈 시스템

- `import`와 `export` 키워드를 사용해 다른 파일의 코드를 가져오거나 내보냄.
- **정적 분석:** 코드가 실행되기 전에 "어떤 모듈이 어떤 모듈을 참조하는지" 컴퓨터(컴파일러, 번들러 등)가
  미리 분석할 수 있음. → 잘못된 경로명 같은 에러를 실행 전에 잡아낼 수 있어 안전성이 높음.
- **비동기 로드:** CommonJS와 비교해 모듈을 가져올 때 비동기로 처리하도록 변경되어 더 효율적임.

* **Node.js에서의 활용:** Node.js는 기본적으로 이전 방식인 CommonJS(`require`)로 동작하기 때문에,
  ES Module 방식을 사용하려면 `package.json` 파일에 `"type": "module"`을 명시적으로 설정해야 함.

**TypeScript**

= 마이크로소프트에서 개발한 오픈 소스 언어, 자바스크립트의 모든 문법을 포함하면서
'타입(Type)'이라는 새로운 기능과 엄격한 문법을 추가한 상위 집합(슈퍼셋, Superset) 언어

- 코드를 실행하기 전(컴파일 단계)에 타입 오류를 미리 잡아주어 런타임 에러를 줄이고 안정성을 높여줌.

**프로젝트 아키텍처**

= 코드를 어떤 구조와 규칙으로 배치할 것인지에 대한 전반적인 설계도

- **프로젝트 아키텍처가 중요한 이유**
  : 코드를 마구잡이로 작성하면 당장은 작동하더라도 기능이 추가될수록 코드가 꼬이게 됨 (스파게티 코드). 좋은 아키텍처는 코드 탐색을 쉽게 하고, 팀원 간의 작업 충돌을 막아주며 , 특정 기능 수정 시 다른 곳에 버그가 터지는 '변경 영향 범위'를 통제하여 유지보수와 확장을 쉽게 만들어 줌.

- **Modular Monolith Architecture**
  : 비즈니스 도메인(관심사)에 따라 모듈을 쪼개어 관리하지만, 실제 배포와 동작은 하나의 서버(Monolith)에서 이루어지는 구조. 결제(Payment), 유저(User) 등 기능 중심으로 코드가 모여 있어 탐색이 직관적이고 협업 시 충돌이 적음.

- **Service-Oriented Architecture (Service Layer Pattern)
  :** 모듈 내부를 역할(비즈니스 로직)에 따라 다시 수평적인 계층(Layer)으로 나누는 패턴.<br> > **Controller:** 클라이언트의 요청(Request)을 받아 흐름을 제어하고 응답(Response)을 반환함.<br> > **Service:** 실제 핵심 비즈니스 로직이 들어가는 곳.<br> > **Repository (Data Access):** 데이터베이스(DB)와 직접 소통하며 쿼리를 수행함.
- **MVC 패턴 (Model-View-Controller)**
  : 과거 웹 개발의 표준이었던 패턴으로, 데이터를 다루는 Model, 화면을 보여주는 View, 중간에서 통제하는 Controller로 나눔. 최근처럼 프론트엔드(React, Vue 등)와 백엔드(Node.js API 서버)가 완전히 분리된 환경에서는 View의 비중이 사라져서, 백엔드에서는 Service Layer 패턴을 더 주력으로 사용하는 추세.
- **그 외 다른 프로젝트 구조**
  - 마이크로서비스 아키텍처(MSA): 서비스가 거대해지면 서버 자체를 도메인별로 물리적으로 쪼갬.
  - 헥사고날 아키텍처(Hexagonal Architecture): 외부 의존성을 격리하는 데 집중함.

**비즈니스 로직 (Business Logic)**

= 애플리케이션이 수행해야 하는 실질적인 '규칙'이자 '핵심 기능'.

- 예) "회원가입 시 비밀번호는 암호화해야 한다", "잔고가 부족하면 결제를 튕겨낸다"
- 프로젝트 아키텍처에서 이 비즈니스 로직은 철저하게 **Service Layer**에 캡슐화되어야 함.

**DTO (Data Transfer Object)**

= '데이터를 옮기는 객체', 계층(Layer) 간에 데이터를 주고받을 때 사용하는 바구니 같은 역할

- **데이터 유효성 검증:** 클라이언트가 보낸 여러 데이터를 DTO라는 객체에 담으면서,
  누락된 값은 없는지 형식이 맞는지(이메일 형식 등)를 검사할 수 있음.
- **민감 정보 필터링:** Service에서 Controller로 데이터를 넘길 때, DTO를 거치면
  DB에 있는 '비밀번호' 같은 민감한 정보를 빼고 안전한 정보만 골라서 전달할 수 있음.
- **유지보수와 범용성:** 컨트롤러의 HTTP Request 객체(`req.body`)를 서비스 레이어에 그대로 넘기면
  서비스가 웹 환경에 종속됨. DTO로 변환해서 넘기면
  서비스 함수가 어디서든 재사용 가능한 범용적인 함수가 됨.
