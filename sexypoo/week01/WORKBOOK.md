### 사용자 관련 요구 사항

1. **카카오 소셜 로그인**을 구현 할 예정이다.
2. **회원 탈퇴 기능**이 필요하다.
3. **이름, 닉네임, 전화번호, 성별**이 필요하다.

### 책 관련 요구 사항

1. **사용자가 책 여러 권을 대여**할 수 있다.
2. 책은 **하나의 카테고리**가 있다.
3. 책은 **제목, 설명에 대한 정보**가 필요하다.
4. 책 소개 페이지에 **해시태그**가 붙을 수 있고,
**책 한 권에 해시태그 여러 개**가, **해시태그 하나가 여러 책**에 붙을 수 있다.
5. 사용자가 책 설명 페이지에서 책에 **좋아요**를 누를 수 있다.
6. 책 **카테고리 별로 현재 몇 개의 책이 있는지 집계**가 필요하다.

### 알림 관련 요구 사항

1. 알림은 **공지 관련 알림, 책 반납 시간 임박 알림, 마케팅 알림이 있을 수 있다.**

## ERD 클라우드로 간단 설계

![image.png](attachment:b5481d45-749e-4ad8-9500-cb113eca6eff:image.png)

1. 테이블 이름과 칼럼 이름은 모두 소문자, 단어 구분은 대소문자가 아닌 밑줄로 구분
2. 기본 키를 위해 각 정보 중 유일한 값을 기본키로 설정 X → index 컬럼을 따로 둠! (이름은 id로 통일하는 것이 좋다)
3. 기본 키 타입은 int가 아닌, 서비스 확장을 고려해서 bigint로 하기

### varchar(20)

MySQL에서 최대 20글자의 글자를 의미 (유니코드 기준 20글자)

### created_at, updated_at

나중에 최신 순 정렬 기능 등을 고려한 설계

datetime(6)은 MySQL에서 밀리초 소수점 6자리까지 구분한다는 의미

### status와 inactivate_date

기능 구현 시 회원 탈퇴, 게시글 삭제 등의 연산을 HTTP 메소드 중 Delete로 바로 삭제하는 방법이 있을 수 있음

→ 이를 Hard Delete라고 하고, 지양하는 것이 좋음

회원 정보 같이 곧바로 지워버리는 것이 위험한 엔티티는 바로 Delete 하는 것이 아니라, **일단 비활성 상태로 두고 일정 기간동안 비활성인 경우 자동 삭제가 되도록 설계**

‼️ 이 때 status를 active, inactive 등 enum으로 관리하기 위해 varchar(15)로 설계한 것이고, 0 또는 1로 해도 됨

‼️ 얼마나 오랫동안 비활성 상태인지 확인하기 위해 inactivate_date를 따로 두는 것

### 자동으로는 어떻게 삭제하는걸까? : batch

정해진 시간에 자동으로 실행되는 프로세스

매일 새벽 2시에 자동으로 user 테이블을 검사, inactive 된 이후 7일이 지난 경우 삭제하도록 할 수 있음

→ 이를 Soft Delete 라고 함

❕ 꼭 회원 뿐만이 아니라, 휴지통 기능이 있는 경우도 soft delete를 적용해야 함!

‼️ soft delete는 당연히 HTTP Method 중 Patch에 해당함

## 연관 관계에 대한 고민

### **1️⃣ 사용자가 책을 대여할 때의 관계를 어떻게 표시해야 할까?**

<aside>
💡

한 종류의 책을 여러 사용자가 대여하고, 한 사용자가 여러 종류의 책을 대여할 수 있음

**N : M 관계**

</aside>

**❕ N:M 관계일 때는 가운데에 매핑 테이블을 따로 둬야함**

![image.png](attachment:00b95356-ad2c-4422-ae5d-3359cab1244a:image.png)

가운데 매핑 테이블은 양쪽의 기본 키를 외래 키로 가지고, **각각과 1:N 관계**를 가짐

### 2️⃣ 책과 책 카테고리의 경우 어떻게 관계를 매핑할까?

책에 카테고리는 한 종류

카테고리 하나 당 여러 종류의 책이 관계를 가짐 → 카테고리와 책은 1:N 관계

IF 책 한 종류에 여러 카테고리가 붙을 수 있다면? 매핑 테이블이 필요함

![image.png](attachment:a0d424cf-8001-453c-b933-8db5593948b9:image.png)

### 3️⃣ 책에 붙는 해시태그는 어떻게 할까?

여러개가 한 책에 붙고, 책에 여러 해시태그가 붙기 때문에 N : M 관계

![image.png](attachment:715d5629-0b08-40f4-9d1b-3687e25385a5:image.png)

### 4️⃣ 책에 붙는 좋아요는 어떻게 할까?

한 종류의 책에 여러명의 사용자가 좋아요를 누르고, 한 사용자가 여러 책에 좋아요를 누름 → N : M 관계

# 🎯 핵심 키워드

### Foreign key 외래 키

두 개의 데이터베이스를 잇고자 기본 키와 함께 사용

- 한 테이블의 필드로, 다른 테이블의 기본 키를 나타냄
- 외래 키를 가지고 있는 테이블을 하위 테이블, 기본 키를 가지고 있는 테이블을 상위 테이블이라고 함
- 각 외부 키와 기본 키는 1:1 매칭되어야 함 (기본 키에 존재하지 않는 값이 외부키로 존재할 수 없음)

### Primary key 기본 키

테이블 내의 모든 레코드에서 고유한 아이디를 제공

- 모든 레코드가 서로 다른 값을 가짐 → 각 레코드를 식별할 수 있게 함
- NULL일 수 없음
- 레코드의 값은 중복 불가능

### ERD : 개체 관계도

ER 모델 (개체-관계 모델, Entity-Relationship Model)

현실 세계에 존재하는 데이터와 그들 간의 관계를 사람이 이해할 수 있는 형태로 표현하기 위해 사용되고 있는 모델

### Composite Key 복합 기본키

두 개 이상의 칼럼으로 구성된 기본 키

데이터베이스 테이블 내에서 레코드의 고유성을 보장

### 연관관계

두 개 이상의 테이블 간에 존재하는 관계

- 1 to 1
- 1 to N
- N to 1
- N to M

네 가지 종류 존재

### 정규화

DBMS 테이블을 최적화하여 정규 형식, 즉 테이블 내에서 속성이 구성되는 방식을 제어하는 규칙 집합

- 데이터의 일관성, 최소한의 데이터 중복, 최대한의 데이터 유연성을 위함
- 중복을 최소화하기 위해 구조화하는 작업
- 하나의 종속성이 하나의 릴레이션에 표현될 수 있도록 분해하는 과정

<aside>
☝

정규화를 하면 불필요한 데이터를 입력하지 않아도 되기 때문에 중복 데이터가 제거됨

</aside>

### 정규화 종류 : 제1정규화

테이블의 칼럼이 원자값 (Atomic value, 하나의 값)을 갖도록 테이블을 분해하는 것

→ 한 칼럼에는 하나의 값만 있어야 함

### 정규화 종류 : 제2정규화

제1정규화를 진행한 테이블에 대해 완전 함수 종속을 만족하도록 테이블을 분해하는 것

<aside>
☝

완전 함수 종속이란?

- 기본키의 부분집합이 결정자가 되어선 안된다는 것을 의미
</aside>

### 정규화 종류 : 제3정규화

제2정규화를 진행한 테이블에 이행적 종속을 없애도록 테이블을 분해하는 것

<aside>
☝

이행적 종속이란?

A→B, B→C가 성립할 때 A→C가 성립되는 것을 의미

- 일반 속성이 다른 일반 속성에 종속되는 것
</aside>

### 반정규화

의도적으로 정규화 원칙을 위배하는 행위

데이터베이스의 성능 향상을 위해 데이터 중복을 허용하고 조인을 줄이는 데이터베이스 성능 향상 방법

조회 속도를 향상시키지만 데이터 모델의 유연성은 낮아짐

<aside>
☝

반정규화 대상

- 수행 속도가 많이 느린 경우
- 테이블의 조인 연산을 지나치게 사용하여 데이터를 조회하는 것이 기술적으로 어려운 경우
- 테이블에 많은 데이터가 있고, 다량의 범위 혹은 특정 범위를 자주 처리해야 하는 경우
</aside>
