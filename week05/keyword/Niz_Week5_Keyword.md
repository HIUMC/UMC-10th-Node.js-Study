# Niz_Week5_Keyword

### 1. 환경 변수 (Environment Variables)

- **정의:** 운영 체제나 실행 환경 레벨에서 어플리케이션에 전달하는 동적인 값들의 모음.
(예: `DB_PASSWORD=1234`, `PORT=8080`)
- **특징:** 코드 외부에 존재하며, 애플리케이션이 실행될 때 로드됨.
주로 `.env` 파일에 작성하고 프레임워크를 통해 불러옴.
- **장단점:**
    - **장점 (Pros):** **보안과 유연성**. 소스 코드에 민감한 정보(API 키, DB 비밀번호 등)를 하드코딩하지 않아도 됨. → 개발(Dev) / 테스트(Test) / 운영(Prod) 환경마다 코드 수정 없이 변수값만 바꿔서 실행할 수 있음.
    - **단점 (Cons):** 변수가 너무 많아지면 관리가 복잡해지고, 팀원 간 동기화(어떤 변수가 추가되었는지 등)가 누락되면 실행 에러가 발생하기 쉬움.
- 주의할 점: 절대 `.env` 파일을 Github에 올리지 말 것!

### 2. CORS (Cross-Origin Resource Sharing)

- **정의:** 서로 다른 출처(Origin: 프로토콜+도메인+포트)를 가진 애플리케이션 간에 리소스를 안전하게 주고받을 수 있도록 허용해 주는 브라우저의 보안 메커니즘.
- **특징:** 브라우저는 기본적으로 '동일 출처 정책(SOP)'을 따름. 이를 우회하여 다른 도메인의 API를 호출하려면, 서버 측에서 HTTP 헤더에 해당 출처를 명시적으로 허용해 주어야 함.
- **장단점:**
    - **장점:** 악의적인 웹사이트가 사용자의 브라우저를 통해 다른 웹사이트의 데이터를 탈취하는 것을 막아주는 강력한 보안 방패 역할을 함.
    - **단점:** 프론트엔드와 백엔드를 분리해서 개발할 때 끊임없이 에러가 발생할 수 있고, 초기 설정이 까다로울 수 있음.
- 주의할 점: 지난 9기 프로젝트 때 프론트엔트로써 CORS 에러 때문에 굉장히 많은 애를 먹은 기억이 납니다… 저희끼리 해결을 해보려 했지만 근본적으로 ‘브라우저가 서버에게 물어보고 차단하는 것’이기 때문에 백엔드에서 해결을 해 주어야 해서 김 빠졌던 기억이 있어요 ㅜㅜ

### 3. DB Connection & DB Connection Pool

- **정의:**
    - **DB Connection:** 애플리케이션(Node.js, Spring 등)과 데이터베이스 간의 물리적인 통신 통로(TCP 세션).
    - **Connection Pool:** 이 Connection들을 매번 새로 만들지 않고, 미리 여러 개를 만들어 '수영장(Pool)'에 보관해 두고 필요할 때마다 빌려주고 반납받는 시스템.
- **특징:** DB 연결은 인증, 네트워크 핸드셰이킹 등 비용(시간과 리소스)이 매우 많이 드는 작업임.
- **장단점:**
    - **장점:** Pool을 사용하면 트래픽이 몰려도 연결을 재사용하므로 **응답 속도가 압도적으로 빨라지고**, DB 서버가 허용치 이상의 연결로 뻗어버리는 것을 막아줌.
    - **단점:** Pool 크기(Size)를 너무 크게 잡으면 오히려 메모리를 낭비하고 DB에 무리를 주며, 너무 작게 잡으면 사용자들이 병목 현상(대기열)을 겪게 됨.
- 주의할 점: 최적의 Pool Size는 CPU 코어 수와 디스크 성능에 비례하며, 생각보다 아주 작기 때문에 얼마나 빨리 처리하고 반납하느냐가 핵심이 됨.

### 4. 비동기 (async / await)

- **정의:** JavaScript/TypeScript에서 비동기 작업(네트워크 요청, 파일 읽기 등 시간이 걸리는 작업)을 동기적인 코드 흐름처럼 직관적으로 작성할 수 있게 해주는 문법.
- **특징:** 내부적으로는 Promise를 기반으로 동작하며, `await` 키워드를 만나면 해당 Promise가 해결(resolve/reject)될 때까지 기다렸다가 다음 줄을 실행함.
- **장단점:**
    - **장점:** 과거의 콜백 지옥(Callback Hell)이나 복잡한 `.then()` 체이닝을 해결하여, 코드의 가독성과 유지보수성을 비약적으로 높여줌.
    - **단점:** `async` 함수는 항상 Promise를 반환한다는 사실을 잊기 쉽고, `await`를 누락하면 데이터가 오기도 전에 다음 코드가 실행되어 `undefined` 에러를 만날 수 있음.
- 주의할 점: 서로 의존성이 없는 작업이라면 `Promise.all()`을 사용해 병렬 처리하기.

### 5. try / catch / finally

- **정의:** 프로그램 실행 중 발생할 수 있는 예외(에러)를 안전하게 처리하기 위한 제어문.
- **특징:**
    - `try`: 실행할 코드 블록.
    - `catch`: 에러가 발생했을 때 실행되는 블록.
    - `finally`: 에러 발생 여부와 상관없이 **무조건** 마지막에 실행되는 블록.
- **장단점:**
    - **장점:** 에러 때문에 프로그램 전체가 다운되는 현상을 막고, 사용자에게 에러 메시지를 보여줌.
    - **단점:** 너무 남발하면(catch 블록 비워두기) 실제 버그를 추적하기 매우 힘들어짐.
- 주의할 점: 주로 리소스 정리 시 `finally` 사용.

### 6. Interface (인터페이스)

- **정의:** (TypeScript 기준) 객체의 '형태'를 정의하는 규약 또는 계약서. 이 객체가 어떤 프로퍼티와 메서드를 가져야 하는지 강제함.
- **특징:** TypeScript는 '덕 타이핑(Duck Typing, 구조적 타이핑)'을 따르기 때문에, 객체가 인터페이스에 정의된 속성만 다 갖추고 있다면 그 인터페이스를 구현한 것으로 인정함.
- **장단점:**
    - **장점:** IDE(VSCode 등)의 강력한 자동완성 기능을 제공받을 수 있고, 컴파일 단계에서 타입 에러를 잡아내어 런타임 에러를 방지함. 코드의 구조를 한눈에 파악하기도 유용함.
    - **단점:** 너무 모든 것에 인터페이스를 억지로 끼워 맞추려다 보면, 코드가 장황해지고 수정이 번거로워질 수 있음.

### 7. Type Assertion (as 키워드)

- **정의:** TypeScript 컴파일러에게 특정 타입 취급을 강제하는 문법입니다.
(예: `const myCanvas = document.getElementById('canvas') as HTMLCanvasElement;`)
- **특징:** 타입 캐스팅(데이터 구조 자체를 변환하는 것)과는 다름. 실제 데이터의 모양을 바꾸는 것이 아니라, 컴파일러의 '눈가림'을 하는 것에 가까움.
- **장단점:**
    - **장점:** DOM 요소를 선택하거나, 외부 API에서 넘어오는 명확하지 않은(any 등) 데이터를 다룰 때 유연하고 빠르게 대처할 수 있음.
    - **단점:** TypeScript의 가장 큰 장점인 **'타입 안정성'이 무너짐**. 잘못 단언하면 컴파일러는 조용히 넘어가지만, 실제 실행 시(런타임) 치명적인 에러가 발생함.
