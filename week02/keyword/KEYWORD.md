## 인덱스 index

추가적인 쓰기 작업과 저장 공간을 활용하여 **데이터베이스 테이블의 검색 속도를 향상시키기 위한 자료구조**

책의 색인

### 인덱스의 장점

데이터를 조회하는 SELECT 외에도 UPDATE나 DELETE의 성능이 함께 향상됨

→ 해당 연산을 위해서는 조회가 필수적이기 때문

### 인덱스의 관리

DBMS는 인덱스를 항상 최신의 정렬된 상태로 유지해야 함 (빠르게 탐색하기 위해)

인덱스가 적용된 컬럼에 다음과 같은 연산이 추가적으로 필요함

- INSERT : 새로운 데이터에 대한 인덱스를 추가
- DELETE : 삭제하는 데이터의 인덱스를 사용하지 않는다는 작업을 진행
- UPDATE : 기존의 인덱스를 사용하지 않음 처리하고, 갱신된 데이터에 대해 인덱스를 추가

### 인덱스의 장점과 단점

- 장점
    - 테이블을 조회하는 속도와 성능을 향상시킬 수 있음
    - 전반적인 시스템 부하를 줄일 수 있음
- 단점
    - 인덱스 관리를 위해 DB의 10%에 해당하는 저장공간이 필요함
    - 인덱스를 관리하기 위해 추가 작업이 필요함
    - 인덱스를 잘못 사용하면 오히려 성능이 저하될 수 있음

### 인덱스를 사용하면 좋은 경우

- 규모가 큰 테이블
- INSERT, UPDATE, DELETE가 자주 발생하지 않는 컬럼
- JOIN이나 WHERE 또는 ORDER BY에 자주 사용되는 컬럼
- 데이터의 중복도가 낮은 컬럼

## 인덱스의 자료구조

### 해시 테이블

(Key, Value) 쌍을 사용하여 컬럼의 값으로 생성된 해시를 통해 인덱스를 구현하는 것

- DB 인덱스에서 사용되는 경우는 제한적 - 해시가 등호 연산에만 특화되어 있기 때문
- 부등호 연산이 자주 사용되는 데이터베이스 검색을 위해서는 적합하지 않음
    - ex) ‘나는’으로 시작하는 모든 데이터를 검색하기 위한 쿼리문은 인덱스 혜택을 받지 못함

### B+Tree

DB의 인덱스를 위해 자식 노드가 2개 이상인 B-Tree를 개선시킨 자료구조

- 리프노드(데이터노드)만 인덱스와 함께 데이터를 가지고 있고, 나머지 노드(인덱스노드)들은 데이터를 위한 인데스(Key)만을 가지고 있음
- 리프노드들은 Linked List로 연결되어 있음
- 데이터 노드 크기는 인덱스 노드 크기와 같지 않아도 됨

## ORM Object Relational Mapping

객체지향 프로그래밍적 사고방식과 관계형 데이터베이스 사이를 연결할 계층의 역할로 제시됨

백엔드 개발자가 DB에 접근하기 위해서는 SQL Query (Raw Query)를 직접 만들어서 보냈어야 했음

→ 백엔드 개발자는 SQL 문법을 알아야 함

→ 개발 프로그래밍과 DB 쿼리코드가 서로 종속됨

### ORM 사용의 장점

- 프로그래밍의 생산성 향상
- 눈에 잘 띄는 쿼리 : 코드가 간결해짐
- DB 쿼리 의존성 감소 : 특정 DB에 종속적이지 않음

### ORM 사용의 단점

- 수정의 어려움 : 자세하지 않아서 수정이 상대적으로 어려움
- 복잡한 쿼리작성의 문제 : ORM이 지정한 명령만 가능, SQL의 문법보다는 없는 문법이 많음
- Raw Query에 비해 느림
