## 📌 SQL의 논리적 실행 순서 (Logical Query Processing Order)

1. **FROM / JOIN**  
   → 어떤 테이블에서 데이터를 가져올지 결정

2. **WHERE**  
   → 조건에 맞는 행만 필터링

3. **GROUP BY**  
   → 남은 행들을 그룹으로 묶음

4. **HAVING**  
   → 그룹 결과 중 조건에 맞는 그룹만 남김

5. **SELECT**  
   → 최종적으로 보여줄 컬럼 선택 및 집계 수행  
   → `COUNT(*)`, `SUM()`, `AVG()` 등 집계 함수 실행

6. **ORDER BY**  
   → 결과 정렬

7. **LIMIT / OFFSET**  
   → 출력할 데이터 개수 제한 (페이징)

---

## 📌 서브쿼리 vs CTE

보통은 **CTE를 더 많이 선호하는 편**이다.

### ✔️ 언제 CTE를 쓰는가

- 가독성이 중요할 때
- 단계적으로 쿼리를 나눠서 생각해야 할 때
- 디버깅이나 수정이 자주 발생할 때

```sql
WITH temp AS (
    SELECT ...
)
SELECT ...
FROM temp;
````

👉 중간 결과에 이름을 붙여서 **읽기 쉬운 구조**로 만들 수 있다.

---

### ✔️ 장점

* 쿼리 흐름이 한눈에 보임
* 복잡한 집계나 분석에 유리
* 단계적으로 나눠서 생각 가능
* 유지보수가 쉬움

---

### ✔️ 예시

```sql
WITH member_like_count AS (
    SELECT
        m.id,
        m.name,
        COUNT(*) AS like_count
    FROM member m
    JOIN book_likes l ON m.id = l.member_id
    GROUP BY m.id, m.name
)
SELECT *
FROM member_like_count
WHERE like_count >= 3;
```

👉 구조가 나뉘어 있어서 읽기가 훨씬 편하다.

---

## ✔️ 언제 Subquery를 쓰는가

* 간단한 조건일 때
* 한 번만 쓰고 끝나는 경우
* `IN`, `EXISTS` 같은 조건문

```sql
SELECT name
FROM member
WHERE id IN (
    SELECT member_id
    FROM book_likes
);
```

👉 굳이 CTE로 뺄 필요 없는 간단한 케이스

---

## ✔️ 실무 기준

* 조건 필터 (`IN`, `EXISTS`) → Subquery
* 집계 / 분석 / 단계적 처리 → CTE

---

## ✔️ 성능

CTE가 무조건 빠르거나, Subquery가 무조건 빠른 건 아니다.
대부분 DB 옵티마이저가 비슷하게 처리한다.

👉 결국 **성능보다 가독성을 먼저 고려하는 게 좋다.**

---

## ✔️ 한 줄 정리

* 단순 조건 → Subquery
* 복잡한 로직 → CTE

---

## 📌 OFFSET 문법

```sql
SELECT *
FROM book
ORDER BY likes DESC
LIMIT y OFFSET (x - 1) * y;
```

### ✔️ 예시

```sql
-- 1페이지
LIMIT 15 OFFSET 0

-- 2페이지
LIMIT 15 OFFSET 15

-- 3페이지
LIMIT 15 OFFSET 30
```

## 📌 데이터베이스 정규화

### ✔️ 정규화란

데이터 중복과 이상 현상을 제거하기 위해
테이블을 분해하는 과정

---

### ✔️ 목적

* 중복 제거
* 삽입 / 수정 / 삭제 이상 방지
* 데이터 일관성 유지
* 저장 공간 효율화

---

### ✔️ 이상 현상

* 삽입 이상 → 불필요한 데이터 없이는 추가 불가
* 수정 이상 → 한 값 수정 시 여러 곳 수정 필요
* 삭제 이상 → 하나 삭제했는데 중요한 정보도 같이 삭제됨

---

## 🟡 1NF (제1정규화)

* 원자성
* 하나의 컬럼에 하나의 값만

---

## 🟡 2NF

* 기본키에 완전 종속
* 부분 종속 제거

->  복합키에서 일부만으로 값이 결정되면 분리

---

## 🟡 3NF

* 이행적 종속 제거

->  A → B → C 구조를 분해

---

## 🟡 BCNF

* 모든 결정자가 후보키가 되도록

---

## 🟡 4NF

->  **서로 독립적인 값들이 한 테이블에 같이 있는 경우 분리**

### 문제 예시

| 학생 | 취미 | 언어  |
| -- | -- | --- |
| A  | 축구 | 영어  |
| A  | 축구 | 일본어 |
| A  | 농구 | 영어  |
| A  | 농구 | 일본어 |

-> 취미와 언어는 서로 독립인데
모든 조합이 만들어짐

---

### 해결

**학생-취미**

| 학생 | 취미 |
| -- | -- |
| A  | 축구 |
| A  | 농구 |

**학생-언어**

| 학생 | 언어  |
| -- | --- |
| A  | 영어  |
| A  | 일본어 |

---

핵심: **불필요한 조합을 제거**

---

## 🟡 5NF

* 조인을 해야 의미가 생기는 경우 분해
* 실무에서는 거의 사용하지 않음

---

## 🔹 반정규화

:  성능을 위해 일부러 중복 허용

### ✔️ 특징

* 조회 성능 향상
* JOIN 감소
* 대신 데이터 일관성 관리 필요

---

### ✔️ 사용 상황

* BI / 대시보드
* 로그 분석
* 데이터 웨어하우스

---

## 📌 인덱스 (Index)

-> 검색 속도를 높이기 위한 자료구조

* 데이터 위치 정보를 따로 저장
* 전체 탐색 없이 빠르게 조회 가능

---

### ✔️ 대표 구조

* Hash Table
* B+Tree

---

## 📌 ORM vs Raw SQL

### ✔️ ORM

* 객체로 DB를 다룸
* 생산성 높음

```ts
const user = await userRepository.find({
  where: { age: 20 }
});
```

---

### ✔️ Raw SQL

* 직접 SQL 작성
* 제어권 높음

```sql
SELECT *
FROM users
WHERE age = 20;
```

---

## ✔️ 차이 정리

| 구분     | ORM | Raw SQL |
| ------ | --- | ------- |
| 추상화    | 높음  | 없음      |
| 제어권    | 낮음  | 높음      |
| 생산성    | 높음  | 낮음      |
| 성능 최적화 | 제한적 | 자유      |
| 복잡 쿼리  | 어려움 | 강력      |

---

## ✔️ 언제 무엇을 쓰는가

* 기본 CRUD → ORM
* 복잡한 쿼리 / 분석 → Raw SQL

---

## ✔️ 핵심 개념

ORM도 결국 SQL을 생성한다

---

## ✔️ 실무 스타일

**ORM + Raw SQL 혼용이 일반적**

---

## 🚀 한 줄 정리

* 빠른 개발 → ORM
* 데이터 처리 / 분석 → SQL
