# Niz_Week3_Keyword

**REST API
(**REST = Representational State Transfer)
→ 인터넷(HTTP)을 기반으로 하는 웹 서비스 아키텍처
→ 클라이언트와 서버가 HTTP 메소드와 자원(Resource)을 이용해서 서로 데이터를 주고받는 통신 규칙

**HTTP 메소드** (CRUD 작업과의 연결)
**> POST (Create - 생성):**
   새로운 데이터를 만들 때 사용.
   클라이언트가 서버로 특정 정보를 넘기면서 처리를 요청하는 경우에도 범용적으로 POST 사용.
> **GET (Read - 조회):**
   데이터베이스에 저장된 데이터를 읽어오고(조회) 싶을 때 사용.
> **PUT / PATCH (Update - 갱신):**
   기존 데이터를 수정할 때 사용. 
   **PUT -** 데이터 전체를 완전히 덮어씌울 때, **PATCH -** 데이터의 일부(예: 닉네임만 변경)만 수정할 때 사용.
> **DELETE (Delete - 삭제):**
   데이터를 삭제할 때 사용.

**데이터 전달 방식 
1. Path Variable (경로 변수)**
   - 목적: 단 하나 존재하는 특정 대상(리소스)을 명확하게 지목할 때 사용.
   - 사용법: 주소(URI)에 직접 식별 값을 넣음. 실제 API 명세서에서 `{}`(중괄호)로 감싸서 변수 표시.
   - 예시: `GET /users/articles/{articleId}`
**2. Query String (쿼리 스트링)**
   - 목적: 데이터가 1개일지 여러 개일지 모르는 상황에서 검색, 필터링, 페이징을 할 때 주로 사용.
   - 사용법: API 주소 끝에 `?`를 붙이고 `키=값` 형태로 전달하며, 여러 개일 때는 `&`로 연결.
                   쿼리 스트링은 API 엔드포인트 자체에는 포함되지 않는다는 점을 주의해야 함.
   - 예시: `GET /users/articles?name=umc&owner=niz`
3**. Request Body (요청 본문)**
   - 목적: `POST`처럼 데이터를 새로 생성하거나 수정할 때, 길고 복잡한 데이터를
               안전하게 서버로 넘기기 위해 사용. 주소에 노출되지 않아 안전함.
   - 사용법: 주로 JSON 형태나 form-data 형태로 데이터를 담아 보냄.
   - 예시: `{ "name": "김연희", "phoneNum": "010-2004-0612", "nickName": "niz" }` 
4**. Request Header (요청 헤더)**
   - 목적: 서버와 데이터를 주고받을 때 필요한 부가적인 정보(메타데이터)를 담는 공간.
   - 사용법: 본문에 담긴 데이터가 JSON인지 알려주거나,
                   현재 로그인된 사용자임을 인증하는 토큰(Token) 등을 담아 보낼 때 사용.
   - 예시: `Authorization: accessToken (String)`
